# Feature Specification: 経路トラッカー (Route Tracker)

**Feature Branch**: `001-route-tracker`
**Created**: 2026-03-20
**Status**: Draft
**Input**: User description: "SwiftUIで直近に通った道をトラックできるアプリ。お酒を飲んでものをなくした時に歩いてきた道を確認できるようにしたい。"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - バックグラウンドでの自動経路記録 (Priority: P1)

ユーザーはアプリを一度起動して記録を開始すれば、その後アプリを操作しなくても移動経路が自動的にバックグラウンドで記録され続ける。お酒を飲んでいる最中にアプリ操作を意識する必要がない。

**Why this priority**: アプリのコア価値。飲酒中はアプリを操作する余裕がないため、意識せずとも記録が継続されることが最も重要。

**Independent Test**: アプリを起動して記録開始後、アプリをバックグラウンドにして数分間歩き回り、再度アプリを開いた時に移動経路が記録されていることを確認する。

**Acceptance Scenarios**:

1. **Given** アプリで記録を開始した状態, **When** アプリをバックグラウンドにして移動する, **Then** 移動経路がバックグラウンドで記録され続ける
2. **Given** バックグラウンド記録中, **When** 端末を再起動する, **Then** 再起動後も記録が自動的に再開される
3. **Given** バックグラウンド記録中, **When** 長時間（数時間）移動し続ける, **Then** バッテリー消費を抑えつつ経路が途切れず記録される

---

### User Story 2 - 経路の地図表示と確認 (Priority: P1)

ユーザーは記録された経路を地図上で視覚的に確認できる。歩いてきた道が線として表示され、どの道を通ったかが一目でわかる。

**Why this priority**: 記録した経路を確認できなければ、落とし物を探す手がかりにならない。記録と表示はセットで最重要。

**Independent Test**: 記録済みの経路データがある状態で地図画面を開き、経路が正しく地図上に描画されていることを確認する。

**Acceptance Scenarios**:

1. **Given** 経路が記録されている状態, **When** 地図画面を開く, **Then** 直近の経路が地図上に線として表示される
2. **Given** 地図に経路が表示されている状態, **When** 地図をズームイン・アウトする, **Then** 経路線が途切れず、地図の縮尺に比例した太さで連続的に表示される
3. **Given** 複数日の経路が記録されている状態, **When** 特定の日付・時間帯を選択する, **Then** 該当する時間帯の経路のみが表示される

---

### User Story 3 - 滞在スポットの可視化 (Priority: P2)

長い時間滞在した場所（飲食店、コンビニなど）が地図上で強調表示される。落とし物をした可能性が高い場所の手がかりになる。

**Why this priority**: 滞在場所は落とし物の可能性が高いスポット。経路だけでなく「どこに長くいたか」がわかることで探索の優先度をつけられる。

**Independent Test**: 特定の場所に10分以上滞在したデータがある状態で、地図上にその滞在スポットが強調マーカーとして表示されることを確認する。

**Acceptance Scenarios**:

1. **Given** 同じ場所に一定時間以上滞在した記録がある, **When** 地図画面で経路を確認する, **Then** 滞在スポットが通常の経路線とは異なる強調表示で表示される
2. **Given** 滞在スポットが地図上に表示されている, **When** 滞在スポットをタップする, **Then** 滞在時間と到着・出発のおおよその時刻が表示される

---

### User Story 4 - 経路履歴の管理 (Priority: P3)

ユーザーは過去の経路履歴を一覧で確認し、不要な記録を削除できる。プライバシーを守りながら必要な記録だけを保持できる。

**Why this priority**: 位置情報はセンシティブなデータ。不要になった記録を自分でコントロールできることが重要。

**Independent Test**: 経路履歴一覧画面から過去の記録を選択・閲覧・削除できることを確認する。

**Acceptance Scenarios**:

1. **Given** 複数日分の経路が記録されている, **When** 履歴一覧画面を開く, **Then** 日付ごとに経路記録が一覧表示される
2. **Given** 履歴一覧が表示されている, **When** 特定の記録を削除する, **Then** その記録が完全に端末から削除され復元できない
3. **Given** 履歴一覧が表示されている, **When** 全データ削除を選択する, **Then** 確認ダイアログの後、すべての経路データが端末から完全に削除される

---

### Edge Cases

- **EC-001**: 地下鉄や地下街など、GPS信号が弱い場所ではどうなるか？ → 信号が弱い区間は点線や異なる表示で「精度が低い区間」として示す
- **EC-002**: バッテリーが極端に低下した場合はどうなるか？ → バッテリー残量が20%以下になった場合、記録頻度を下げて省電力モードで継続する
- **EC-003**: 位置情報の権限が途中で取り消された場合は？ → 記録を一時停止し、アプリ再開時に権限の再設定を促す通知を表示する
- **EC-004**: 長距離移動（電車・車）で大量のデータが生成された場合は？ → 移動速度に応じて記録間隔を自動調整し、データ量を最適化する
- **EC-005**: 記録が数日分溜まった場合のデータ量は？ → 古いデータの自動アーカイブまたは解像度の低減で端末ストレージを保護する
- **EC-006**: アプリがユーザーによって強制終了（スワイプキル）された場合は？ → 記録を停止し、中断前までのデータを保持する。次回アプリ起動時に「記録が中断されました」と通知し、記録の再開を促す
- **EC-007**: 経路データが存在しない状態（初回起動時）は？ → 現在地を中心とした地図を表示し、「記録を開始」の目立つCTAボタンと簡易オンボーディング（1画面）で操作を案内する

## Out of Scope (v1)

- クラウド同期・バックアップ
- データエクスポート（GPX等の外部形式への書き出し）
- 他ユーザーとの経路共有・ソーシャル機能
- Apple Watch 対応
- マルチデバイス間のデータ同期

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: システムはアプリがバックグラウンドにある状態でも位置情報を継続的に記録できなければならない [US-1]
  - **FR-001a**: ユーザーは経路記録の開始と停止を明示的に操作できなければならない [US-1]
  - **FR-001b**: システムは記録中であることをユーザーに視覚的に示さなければならない（ステータス表示） [US-1]
- **FR-002**: システムは記録した経路を地図上に線として描画できなければならない [US-2]
  - **FR-002a**: ユーザーは日付・時間帯を指定して過去の経路を閲覧できなければならない [US-2, US-4]
- **FR-003**: システムは半径50メートル以内の同一地点に一定時間（デフォルト5分）以上滞在した場合、その場所を滞在スポットとして自動検出しなければならない [US-3]
- **FR-004**: システムは滞在スポットを地図上で通常の経路線とは視覚的に区別して強調表示しなければならない [US-3]
- **FR-005**: システムはすべての位置情報データを端末のローカルストレージにのみ保存しなければならない [US-1, US-4]
- **FR-006**: システムは外部サーバーへの位置情報データの送信を行ってはならない [US-1, US-4]
- **FR-006a**: システムは保存された位置情報データをiOS Data Protection CompleteUntilFirstUserAuthentication（NSFileProtectionCompleteUntilFirstUserAuthentication）で暗号化しなければならない。データは端末の再起動後、初回ロック解除以降アクセス可能となり、バックグラウンド記録中の書き込みと暗号化保護を両立する [US-1, US-4]
- **FR-007**: ユーザーは個別の経路記録または全データを完全に削除できなければならない [US-4]
- **FR-008**: システムはGPS信号の強度に応じて、経路ポイントごとに精度情報を保持しなければならない [US-2]
- **FR-009**: システムはGPS精度が低い区間を、通常の経路線とは視覚的に区別して表示しなければならない（例: 点線表示） [US-2]
- **FR-010**: システムはバッテリー消費を抑えるため、移動速度・加速度・GPS精度に応じて位置情報の取得頻度を自動調整しなければならない。調整ルールは以下の通り: [US-1]
  - 静止時（速度 < 1 km/h）: 60秒間隔
  - 歩行時（1〜6 km/h）: 10秒間隔
  - 走行・自転車（6〜30 km/h）: 5秒間隔
  - 車両・電車（> 30 km/h）: 30秒間隔
  - バッテリー残量20%以下の省電力モード: 上記間隔を2倍に延長
- **FR-011**: システムはデータの肥大化を防ぐため、30日以上経過した経路データを自動的に削除しなければならない。ストレージ使用量が100MBを超えた場合は、古いデータから優先的に削除する [US-4]

### Key Entities

- **経路記録 (Route Record)**: 特定の期間における一連の位置情報の集合。開始時刻、終了時刻、経路ポイントの集合、総移動距離を持つ
- **経路ポイント (Route Point)**: 個々の位置情報。緯度、経度、記録時刻、精度情報、移動速度を持つ
- **滞在スポット (Stay Spot)**: 一定時間以上同一地点に留まったことを示す情報。中心座標、到着時刻、出発時刻、滞在時間を持つ。経路記録に紐づく

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: ユーザーはアプリ操作なしで、バックグラウンドで途切れることなく最低6時間の経路記録ができる
- **SC-002**: 記録された経路を地図上で確認するまでの操作が、アプリ起動から3タップ以内で完了する
- **SC-003**: 5分以上の滞在スポットが95%以上の精度で自動検出される
- **SC-004**: 6時間のバックグラウンド記録によるバッテリー消費が端末バッテリー全体の15%以下に収まる
- **SC-005**: すべての位置情報データがローカルにのみ保存され、外部への通信が一切発生しないことが検証できる
- **SC-006**: 初めて使うユーザーが記録開始までの操作を30秒以内に完了できる
- **SC-007**: 6時間分の経路データ（最大数万ポイント）の地図描画が2秒以内に完了する

## Assumptions

- 対象プラットフォームはiOS（SwiftUI使用の要求より）
- ユーザーは外出時にiPhoneを携帯している前提
- 位置情報の「常に許可」権限をユーザーが付与する前提
- 滞在スポットの検出閾値はデフォルト5分とし、将来的にユーザーが調整可能にする余地を残す
- データ保持期間はデフォルトで30日間とし、古いデータは自動的にアーカイブまたは削除される
- 地図表示にはiOS標準のMapKitを想定（ただし実装詳細は計画フェーズで決定）

## Clarifications

### Session 2026-03-20

- Q: 端末に保存される位置情報データの保護方式は？ → A: iOS Data Protection CompleteUntilFirstUserAuthentication で暗号化（当初 Complete を検討したが、バックグラウンド記録中の端末ロック時にDB書き込み不可となるため変更）。アプリ独自のロック機能は v1 では不要（iOS端末ロックで代替）
- Q: v1 で明示的に除外するスコープは？ → A: クラウド同期、データエクスポート、他ユーザーとの共有、Apple Watch対応をすべて除外
- Q: アプリが強制終了された場合の記録挙動は？ → A: 記録停止＋中断前データ保持＋次回起動時に中断通知と再開促進
- Q: 経路データが存在しない状態の地図画面表示は？ → A: 現在地中心の地図 + 目立つ「記録を開始」CTAボタン + 簡易オンボーディング（1画面）
- Q: 地図画面での経路描画の許容レスポンス時間は？ → A: 6時間分の経路データで2秒以内（間引き・キャッシュで実現）
